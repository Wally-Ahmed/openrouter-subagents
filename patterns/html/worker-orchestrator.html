Worker Orchestrator Pattern

The no-gate sibling of the two-layer expert pattern: fan concrete implementation work out to the OpenRouter worker (the ask_openrouter tool with a fast model) through cheap Sonnet wrapper subagents — and let execution, not a verification gate, decide it's correct.

Why wrap the worker (the point of the pattern)

ConcernCalling ask_openrouter directly from your main loopWrapping it in Sonnet subagents
Context bloat Every file / stack trace the worker needs is read into your main context. Bundle assembly happens inside each subagent; the main loop only sees the compact result. cheap
Parallelism Calls from the main loop are effectively serial. Subagents run truly concurrently in bounded waves. fast
Cost Orchestrator-model tokens spent shuttling raw inputs. A cheap Sonnet wrapper does the shuttling. cheap
Durability Results live only in the conversation → lost on compaction / restart. Each result written to a durable artifact as it returns. durable
The deliberately absent row is a verification gate. Worker output is a runnable artifact, so it's validated by execution (build / tests) — not trusted as a claim. That's the one intentional difference from two-layer-cross-model-expert, which does verify claims against ground truth. Reach for that one when the output is a judgment you must trust before acting.

The moves

  1. Split the work into independent units (separate files, bugs, refactors).
  2. Bundle — each wrapper gathers a tight, high-signal input set for its unit.
  3. Call ask_openrouter with a precise task + the bundle, choosing a fast model.
  4. Integrate — turn the output into a usable artifact; do a light sanity check.
  5. Validate by execution — build, run tests, apply the patch. A failure becomes a new work unit.
  6. Aggregate — append each unit's result to a durable artifact as it returns.

Worker vs. expert — same tool, different pattern

There's a single ask_openrouter tool. Which role it plays is your choice of pattern, model, and effort:

worker-orchestrator (this)two-layer-cross-model-expert
OutputConcrete artifacts: patches, fixes, tests.Judgments: reviews, critiques, threat models.
CorrectnessChecked by running it.Checked by verifying claims vs ground truth.
WrapperCheap Sonnet, no gate.Heavier wrapper + a verification gate.
ask_openrouter callFast model.Strong model + reasoning_effort: high.

Operating guidelines

  • Bounded waves (~3 concurrent) — avoids MCP / client timeouts and contention.
  • Validate by running, every time — never merge a worker artifact you haven't built / tested.
  • Keep the wrapper cheap (Sonnet). If a unit needs claim-verification (an assertion about the system, not a runnable artifact), switch it to the two-layer pattern.
  • Write results incrementally to a durable artifact so a mid-wave failure loses nothing.
The reusable idea, in one line: fan concrete work out to a cheap wrapper around the OpenRouter worker, and let execution — not a verification gate — be the thing that decides it's correct.
