Two-Layer Cross-Model Expert Pattern

A reusable way to put an external "expert" model (reached over MCP) to work inside an agent loop — wrapping it in orchestrating subagents so its output stays parallel, context-cheap, and trustworthy.

Layer 0 · Orchestrator

Main agent session

Owns the work queue. Splits work into independent units, fans out a bounded wave of wrapper subagents in parallel, and merges their results into one durable artifact as each lands.

Why wrap the expert (the point of the pattern)

ConcernCalling the MCP expert directly from your main loopWrapping it in subagents
Context bloat Every input the expert needs is read into your main context → it fills, compaction kicks in, working state is lost. Bundle assembly happens inside each subagent; the main loop only sees the compact result. cheap
Parallelism Calls from the main loop are effectively serial. Subagents run truly concurrently in bounded waves. fast
Unverified claims Expert output is consumed as-is → false claims get acted on, wasting effort or adding errors. Subagent verifies each claim against ground truth before it surfaces. safety
Shared blind spots If the expert is the same model doing the work, it shares your failure modes. A different model family surfaces issues a self-pass would miss. rigor
Durability Results live only in the conversation → lost on compaction/restart. Each result written to a durable artifact as it returns. durable

The five moves

  1. Split the work into independent units the orchestrator can dispatch without shared state.
  2. Bundle — each wrapper gathers a tight, high-signal set: the material to analyze plus the ground-truth sources to check it against. Read only what's relevant, but give the expert depth, not a summary.
  3. Call the expert over MCP with a pointed question + the bundle as context.
  4. Verify — treat output as hypotheses; check each claim against ground truth and tag it VERIFIED / FALSE / UNCERTAIN.
  5. Aggregate — return a compact result; the orchestrator appends it to a durable artifact as each unit completes.

What goes in each bundle

A reviewer is only as good as its inputs. Give the expert depth — never ask for a "concise" answer when the task is a deep analysis:

① The material

The actual thing to analyze (code, document, design, dataset slice).

② Ground truth

The authoritative sources to check the answer against (specs, requirements, source of record).

③ The ask

A pointed, critical question framing exactly what to evaluate and against what.

Operating guidelines

  • Bounded waves, not unbounded fan-out. Run a small number of expert calls concurrently (e.g. ~3) — massive parallel dispatch tends to hit MCP/client timeouts and contention.
  • Write results incrementally to the durable artifact as they return, so a mid-wave failure never loses completed work.
  • Pick an expert from a different model family than the one building/orchestrating — diverse blind spots are the whole point.
  • Use high reasoning effort on the expert when the task is a deep audit/critique, not a quick lookup.
  • Never skip the verification gate — a confident wrong answer that gets acted on is worse than no answer.

When to reach for it

Use this whenever you want an external / second-opinion model (or any MCP tool prone to confident-but-wrong output) without (1) flooding your main context, (2) serializing your work, or (3) blindly trusting the result. Typical fits: code/security review, design critique, threat modeling, cross-checking research, or analyzing material too large to hold in the main context.

The reusable idea, in one line: wrap an external MCP expert in a subagent that bundles → calls → verifies, so the main loop only ever sees trustworthy, compact conclusions.
